## Key Error Handling Patterns:

**1. HTTP Status Codes via HttpResponseError:**
- **403** - Access denied, check RBAC roles ('Key Vault Secrets User' or 'Key Vault Secrets Officer')
- **404** - Secret not found (could be deleted or never existed)
- **429** - Rate limited, check `Retry-After` header and implement backoff
- **401** - Authentication failed

**2. Inspecting Error Details:**
```python
except HttpResponseError as e:
    e.status_code      # HTTP status code (403, 404, 429, etc.)
    e.message          # Human-readable error message
    e.error.code       # Structured error code
    e.error.message    # Detailed error message
    e.response.headers # Access headers like 'Retry-After'
```

**3. Soft-Deleted Secrets:**
When you try to `get_secret()` on a soft-deleted secret:
- ❌ Returns **404** (just like non-existent secrets)
- ✅ Use `get_deleted_secret()` to check if it's soft-deleted
- You can then recover it with `begin_recover_deleted_secret()`
- You **cannot** create a new secret with the same name until it's recovered or purged

**4. Best Practices:**
- Don't retry 403/404 errors (won't succeed)
- Implement exponential backoff for 429 errors
- Use the `Retry-After` header when provided
- Handle `ServiceRequestError` for network issues

The example file includes practical functions for single operations, bulk operations, and retry logic!
